# 코멘트

## 목적

이 문서는 기존 팀 설계 초안에서 충돌하거나 운영상 취약한 부분을 왜 수정했는지 남기는 기록이다.

## 1. 예시 프로젝트 흔적 제거

- 이유: 기존 요구사항 자료에 다른 프로젝트 흔적이 섞여 있었다.
- 조치: 최종 기준서는 TripZone 전용 용어와 기능만 남기고 재서술했다.

## 1-1. 구조명세 스프레드시트 반영

- 이유: 뒤늦게 확인한 구조명세 스프레드시트가 있었고, 이 안에는 실제 패키지/클래스 뼈대가 들어 있었다.
- 조치: 스프레드시트 내용을 기준으로 `tripzone-structure-spec-v2.md`를 별도로 만들고, 도메인 설계와 맞지 않는 항목만 교정했다.
- 효과: DB 설계와 패키지 설계가 따로 노는 문제를 줄였다.

## 2. 권한 구조를 `USER_ROLES` 기준으로 통일

- 이유: 메모에는 `MEMBER_AUTH`, `PERSISTENT_LOGINS` 같은 과거 이름이 섞여 있었고, 최신 SQL은 `USER_ROLES`, `USER_AUTH_PROVIDERS`, `USER_REFRESH_TOKENS` 쪽으로 정리돼 있었다.
- 조치: 권한은 `USER_ROLES`, 인증수단은 `USER_AUTH_PROVIDERS`, 세션성 토큰은 `USER_REFRESH_TOKENS`로 통일했다.
- 효과: 로그인, 권한, 판매자 승인 책임이 분리된다.

## 3. 문의 구조를 `INQUIRY_ROOMS` + `INQUIRY_MESSAGES`로 변경

- 이유: 초안에는 `INQUIRIES`, `COMMENT`가 있었지만, 팀 대화에서는 문의센터를 채팅형으로 바꾸자는 결정이 있었다.
- 조치: 게시글+댓글 구조 대신 문의방과 메시지 구조로 재설계했다.
- 효과: 실시간 문의, 운영자 개입, 읽음 처리, 문의 상태 관리가 쉬워진다.

## 3-1. 구조명세서의 문의 클래스도 함께 교정

- 이유: 구조명세서에는 `Inquiry`, `Comment`, `CommentRepository`가 남아 있었고, 동시에 `InquiryRoomRepository`도 같이 존재했다.
- 조치: 구조 기준을 `InquiryRoom`, `InquiryMessage`, `InquiryMessageRepository`로 통일했다.
- 효과: 클래스 구조와 DB 구조가 같은 방향을 보게 했다.

## 4. `HOST_PROFILES` 승인 상태 확장

- 이유: `PENDING`, `APPROVED`, `REJECTED`만으로는 운영 중지나 제재 처리가 애매하다.
- 조치: `SUSPENDED` 상태와 `SUSPENDED_AT`을 추가했다.
- 효과: 승인 이후에도 판매자 운영 제어가 가능하다.

## 5. `USERS`에 `USER_STATUS`, `DELETED_AT` 추가

- 이유: 기존 `ENABLED`만으로는 휴면, 차단, 탈퇴 상태를 구분하기 어렵다.
- 조치: `ENABLED`는 유지하고, 운영 상태를 위한 `USER_STATUS`와 `DELETED_AT`을 추가했다.
- 효과: 기존 컬럼 호환성을 유지하면서 운영 상태를 분리할 수 있다.

## 6. 숙소와 객실에 soft-delete 성격 컬럼 추가

- 이유: 숙소와 객실은 예약/리뷰/정산과 연결되므로 물리 삭제가 위험하다.
- 조치: `DELETED_AT`을 추가하고 상태값도 확장했다.
- 효과: 데이터 보존과 운영 복구가 가능하다.

## 7. 예약 상태와 취소 추적 보강

- 이유: 기존 `BOOKINGS`는 취소 이력과 운영자 개입 추적이 약했다.
- 조치: `CANCELED_BY_USER_NO`, `CANCELED_AT`, `NO_SHOW` 상태를 추가했다.
- 효과: 예약 취소 책임 주체와 최종 상태가 명확해진다.

## 8. 결제 제약 강화

- 이유: 환불 금액이 결제 금액보다 커질 수 있는 위험을 막아야 한다.
- 조치: `REFUND_AMOUNT <= PAYMENT_AMOUNT` 체크를 추가했다.
- 효과: 명백한 데이터 오류를 DB에서 1차 차단한다.

## 9. 리뷰 운영 상태 추가

- 이유: 기존 리뷰 테이블은 작성만 있고 신고, 숨김, 운영 통제가 불가능했다.
- 조치: `REVIEW_STATUS`를 추가했다.
- 효과: 운영자가 부적절한 리뷰를 관리할 수 있다.

## 10. 인덱스 추가

- 이유: 팀 초안은 관계는 있었지만 조회 패턴 기준 인덱스가 거의 없었다.
- 조치: 예약, 결제, 숙소, 문의, 쿠폰, 리뷰에 조회용 인덱스를 추가했다.
- 효과: 화면 조회와 운영 목록 성능의 기본선을 확보한다.

## 10-1. 구조명세서 오타 수정

- 이유: 구조명세서에 `BookindController`, `LodgindImage` 같은 오타가 있었다.
- 조치: 개정 구조명세서에서 `BookingController`, `LodgingImage`로 정리했다.
- 효과: 구현 시작 전에 클래스명 혼선을 줄일 수 있다.

## 11. 감사 로그 추가

- 이유: 실무 운영에서는 승인, 차단, 강제 취소 같은 행위를 남겨야 한다.
- 조치: `AUDIT_LOGS`를 추가했다.
- 효과: 운영 추적성과 책임 소재 확인이 가능하다.

## 12. 그대로 둔 것

- `*_NO` 기본키 명명
- `REG_DATE`, `UPD_DATE`
- `MEMBER_GRADES`, `USERS`, `HOST_PROFILES`, `LODGINGS`, `ROOMS`, `BOOKINGS`, `PAYMENTS`, `REVIEWS`, `WISHLISTS`, `MILEAGE_HISTORY` 같은 핵심 테이블명
- 쿠폰/이벤트/마일리지의 전체 도메인 방향

## 13. 아직 서비스 레이어에서 막아야 하는 것

- 관리자 권한 사용자가 실제로만 승인 처리하는지 검증
- 중복 예약 방지 로직
- 숙박 완료 예약만 리뷰 작성 가능하도록 검증
- 회원 쿠폰 재사용 방지 로직
- 문의방 접근 권한 검증

## 14. 최종 판단

이번 개정은 기존 학원형 설계를 버린 게 아니라, 실무에서 설명 가능한 형태로 보강한 상위호환 정리다.
