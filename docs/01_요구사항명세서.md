# TripZone 요구사항 명세서

## 1. 프로젝트 개요

- 프로젝트명: TripZone
- 프로젝트 유형: 숙소 예약 플랫폼
- 목적: 사용자가 국내 숙소를 탐색하고 예약, 결제, 리뷰, 문의까지 처리할 수 있는 통합 서비스 구축

## 2. 사용자 구분

### 2.1 일반 회원

- 회원가입 및 로그인
- 숙소 검색 및 상세 조회
- 예약 및 결제
- 리뷰 작성
- 찜 등록
- 문의 등록
- 쿠폰 및 마일리지 사용

### 2.2 판매자

- 판매자 승인 신청
- 숙소 등록/수정/비노출 처리
- 객실 등록/수정/비노출 처리
- 예약 확인 및 취소 처리
- 문의 응답

### 2.3 관리자

- 회원 관리
- 판매자 승인/반려/중지
- 이벤트/쿠폰 관리
- 문의 모니터링
- 운영 로그 확인

## 3. 핵심 요구사항

### 3.1 회원

- 회원은 이메일 기준으로 고유해야 한다.
- 회원은 일반, 판매자, 관리자 권한을 가질 수 있다.
- 로그인 방식은 일반 로그인 및 소셜 로그인 확장 가능 구조여야 한다.

### 3.2 판매자

- 판매자는 일반 회원 확장 개념이다.
- 판매자는 승인 전까지 숙소 운영 권한이 없다.
- 판매자 상태는 승인 대기, 승인, 반려, 중지로 관리한다.

### 3.3 숙소/객실

- 판매자는 숙소와 객실을 등록할 수 있어야 한다.
- 숙소와 객실은 운영 상태를 가져야 한다.
- 숙소 상세에는 이미지, 설명, 체크인/체크아웃 정보가 포함되어야 한다.

### 3.4 예약/결제

- 예약은 객실 단위로 생성한다.
- 예약 시 숙박일, 인원수, 가격, 할인, 총 결제금액이 저장되어야 한다.
- 결제 상태는 준비, 결제완료, 실패, 취소, 부분취소, 환불을 지원해야 한다.

### 3.5 리뷰

- 리뷰는 숙박 완료 예약에 대해서만 작성 가능해야 한다.
- 한 예약당 한 개의 리뷰만 허용한다.
- 리뷰는 이미지 첨부가 가능해야 한다.

### 3.6 찜

- 회원은 숙소를 찜할 수 있다.
- 동일 회원이 동일 숙소를 중복 찜할 수 없어야 한다.

### 3.7 문의

- 문의는 채팅형 구조를 기준으로 한다.
- 회원은 문의방을 생성하고 메시지를 주고받을 수 있어야 한다.
- 문의 유형은 숙소, 예약, 결제, 시스템 문의를 지원한다.

### 3.8 쿠폰/이벤트/마일리지

- 관리자는 이벤트를 등록할 수 있어야 한다.
- 관리자는 쿠폰을 발급하고 상태를 변경할 수 있어야 한다.
- 회원은 보유 쿠폰과 마일리지를 예약 시 사용할 수 있어야 한다.

## 4. 비기능 요구사항

- 주요 상태 변경은 추적 가능해야 한다.
- 운영용 조회 성능을 위해 주요 검색 조건에 인덱스가 필요하다.
- 삭제는 기본적으로 soft-delete 또는 상태 변경 중심으로 처리한다.
- 데이터 정합성은 DB 제약조건과 서비스 검증을 함께 사용한다.
