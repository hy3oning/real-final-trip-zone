Company Spec

TripZone 통합 기준서

국내 숙소 플랫폼 3 Role 운영 문의 채팅형 기준 DB/구조 기준 통합

이 문서는 팀 내부 초안, 테이블 메모, ERD, 구조 스프레드시트를 현재 구현 흐름에 맞게 하나의 기준으로 정리한 통합 문서다. 공통으로 참고할 원칙과 상태 모델, 우선순위를 함께 담고 있다.

3일반회원, 판매자, 관리자 역할 체계
8회원/판매자/숙소/예약/리뷰/문의/혜택/감사 로그 우선순위
2문의방과 메시지로 구성한 문의 모델

Principles

핵심 설계 원칙

  • 기본키는 NUMBER(10) + 시퀀스 기반으로 유지한다.
  • 시간 컬럼은 REG_DATE, UPD_DATE를 기본으로 사용한다.
  • 회원, 판매자, 숙소, 객실, 쿠폰, 문의방은 즉시 물리 삭제하지 않고 상태값 또는 DELETED_AT 중심으로 관리한다.
  • 권한 기준은 USER_ROLES로 통일하고, 판매자 여부는 HOST_PROFILES와 함께 판단한다.

Domain Model

도메인 모델 요약

회원 / 판매자

  • USERS가 회원 공통 프로필과 상태를 가진다.
  • USER_AUTH_PROVIDERS로 인증수단을 분리한다.
  • HOST_PROFILES에서 판매자 신청과 승인 상태를 관리한다.

숙소 / 객실 / 예약

  • LODGINGS는 숙소 마스터다.
  • ROOMS는 실제 예약 단위다.
  • BOOKINGS는 객실 기준 예약 스냅샷을 보관한다.

결제 / 혜택 / 리뷰

  • PAYMENTS는 활성 결제 기준 레코드를 가진다.
  • USER_COUPONS와 MILEAGE_HISTORY를 함께 사용한다.
  • 리뷰는 숙박 완료 예약에 대해서만 1회 작성 가능하다.

문의

  • 문의는 게시판형이 아니라 채팅형 구조를 기준으로 한다.
  • 문의방: INQUIRY_ROOMS
  • 메시지: INQUIRY_MESSAGES

Integrity

정합성 / 인덱스 기준

  • 이메일 유니크
  • 한 회원은 하나의 HOST_PROFILE만 가진다.
  • 동일 객실의 동일 날짜 범위 중복 예약은 서비스 레이어와 인덱스로 방지한다.
  • 동일 회원/숙소 찜 중복 금지, 동일 예약 리뷰 중복 금지
  • USERS(EMAIL), BOOKINGS(ROOM_NO, CHECK_IN_DATE, CHECK_OUT_DATE, STATUS) 등 주요 검색 인덱스를 둔다.

Priority

구현 우선순위

01
회원 / 권한 / 인증

로그인, 권한, 판매자 확장 구조를 먼저 정리한다.

02
판매자 승인

승인 흐름과 운영 제재 모델을 확정한다.

03
숙소 / 객실 / 예약 / 결제

프론트와 백엔드가 함께 참고하는 핵심 예약 축을 우선 확정한다.

04
리뷰 / 찜 / 문의 / 혜택 / 감사 로그

운영과 사용자 편의 기능을 마무리한다.
