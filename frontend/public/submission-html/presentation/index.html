TripZone Final Presentation
01 개요

Project Thesis

TripZone는 demo 기준이 아니라 설계 기준으로 다시 정리한 팀 프로젝트입니다

기존 데모에서 좋은 시각 요소는 가져오되, 실제 제출물과 발표자료는 설계서 기준으로 다시 구성했다. 특히 패키지명, 도메인 구조, 문의 모델, 상태값 기준을 고정했다.

com.kh.trip 3-role platform InquiryRoom / InquiryMessage

Design-fixed package

설계 기준 고정

4 + 1
명세서 HTML 4종 발표용 deck 1종

Problem

기존 demo와 설계 기준이 어긋나 있었다

  • 현재 구현은 demo 기준으로 발전해 설계 문서와 일부 차이가 있었다.
  • 구조명세와 DB 설계가 서로 다른 시점의 흔적을 동시에 갖고 있었다.
  • 문의 모델, 패키지명, 상태값 기준이 그대로면 발표와 제출에서 혼선이 생긴다.

Approach

그래서 구현물과 설계물을 분리했다

  • trip-demo는 참고 자산으로만 사용
  • 설계 기준 전용 제출물은 새 HTML 세트로 구성
  • 프론트만 우선 구현하고, 백엔드는 최소 고정점만 명시

Scope

TripZone의 제품 범위

숙소 예약 플랫폼을 사용자, 판매자, 관리자 세 역할로 분리해 설계했다.

User

회원가입, 로그인, 숙소 탐색, 예약, 결제, 리뷰, 찜, 문의, 혜택 사용

Seller

판매자 승인 신청, 숙소/객실 등록, 예약 처리, 문의 응답

Admin

회원/판매자 관리, 이벤트/쿠폰 관리, 문의 모니터링, 감사 로그 관리

Roles

핵심 구조는 3-role 분리다

일반 회원

경험 중심 사용자

  • 검색
  • 예약
  • 결제
  • 리뷰
  • 문의

판매자

운영 중심 사용자

  • 숙소 등록
  • 객실 관리
  • 예약 처리
  • 문의 응답

관리자

통제 중심 사용자

  • 판매자 승인
  • 이벤트/쿠폰
  • 문의 모니터링
  • 감사 로그

Structure

루트 패키지명은 고정한다

com.kh.trip.domain com.kh.trip.dto com.kh.trip.repository com.kh.trip.service com.kh.trip.controller

이번 발표와 제출물은 현재 demo 구현 패키지가 아니라 구조명세서 기준 패키지로 설명한다.

주요 도메인

  • User / HostProfile / MemberGrade
  • Lodging / Room / Wishlist
  • Booking / Payment / Review

문의 도메인

  • InquiryRoom
  • InquiryMessage
  • InquiryService / InquiryController

Database

DB는 상태 기반 운영 모델로 보강했다

회원

ACTIVE, DORMANT, BLOCKED, DELETED

판매자

PENDING, APPROVED, REJECTED, SUSPENDED

예약

PENDING, CONFIRMED, CANCELED, COMPLETED, NO_SHOW

핵심 테이블

  • USERS, HOST_PROFILES
  • LODGINGS, ROOMS
  • BOOKINGS, PAYMENTS
  • REVIEWS, WISHLISTS

운영 / 혜택 / 문의

  • EVENTS, COUPONS, USER_COUPONS
  • MILEAGE_HISTORY, AUDIT_LOGS
  • INQUIRY_ROOMS, INQUIRY_MESSAGES

Conflict Fix

가장 큰 충돌은 문의 구조였다

초안에는 Inquiry / Comment가 있었고, 대화에서는 채팅형 문의로 방향이 바뀌었다.

기존 흔적Inquiries / Comment / InquiryRoomRepository 혼재
최종 기준InquiryRoom = 문의방, InquiryMessage = 메시지
효과읽음 처리, 운영자 개입, 상태 관리, 실시간 대화 확장에 유리

Frontend First

이번 구현은 프론트 산출물 중심이다

  • 제출용 명세서 HTML 4종
  • 발표용 PPT HTML 1종
  • 인쇄/PDF 변환 대응 레이아웃

Backend Minimum

백엔드는 최소 고정점만 명시한다

  • 루트 패키지명 고정
  • 도메인명과 상태값 기준 고정
  • 문의 구조와 핵심 엔티티 기준 고정
  • 실제 구현은 팀원 백엔드 작업 범위

Submission

최종 제출물 구성

문서형

  • 요구사항 명세서
  • 기능 명세서
  • 구조 명세서
  • DB 명세서

기준형

  • DDL
  • 수정 사유 코멘트
  • 발표용 HTML deck

Closing

이번 정리는 demo 개선이 아니라 설계 기준 고정 작업이다

발표와 제출은 설계 기준으로, 실제 구현은 이후 팀 작업으로 이어질 수 있게 문서와 화면 기준을 먼저 정리했다.

Key Result

  • 패키지 기준 고정
  • 문의 구조 충돌 해소
  • DB 상태 모델 정리
  • 프론트 제출물과 발표물 완성
