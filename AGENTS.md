# TripZone Working Context

## Project Priority

- 이 프로젝트의 기준은 끝까지 설계도다.
- 참고 UI는 `trip-demo`, Airbnb, 여기어때의 좋은 부분만 선별 반영한다.
- 설계도에 없는 기능은 임의로 추가하지 않는다.
- 설계도에 있는 흐름과 역할은 삭제하지 않는다.

## Role Boundary

- 이 저장소에서 내 역할은 발표 직전 프론트 안정화와 검증 우선이다.
- 백엔드는 팀원이 이어받을 수 있도록 최소 골격과 가이드만 유지한다.
- 백엔드 구현을 확장하지 않는다. 필요한 경우에도 루트, 패키지, README, 최소 연결점까지만 다룬다.

## UI Direction

- 목표는 팀원에게 넘겨도 되는 실서비스급 여행 예약 프론트다.
- UI/UX는 현재 확정본을 기준으로 유지한다.
- 메인은 여행 사이트 메인으로 유지한다.
- 문서 허브, 제출물 허브, 설계 설명 느낌을 사용자 메인에서 전면에 내세우지 않는다.
- 상세, 리스트, 예약, 마이페이지, 판매자/관리자 화면은 신규 리디자인보다 동작 안정성을 우선한다.
- 디자인 기준은 `frontend-skill` 원칙을 따른다.

## Frontend Skill Rules To Keep

- 카드보다 구성과 위계를 먼저 본다.
- 이미지가 실제 역할을 해야 한다.
- 색 포인트는 의미 있는 곳에만 쓴다.
- 카피는 줄이고, 상태/가격/후기/행동을 먼저 보이게 한다.
- 화면이 밋밋하다고 박스와 그림자를 먼저 늘리지 않는다.
- spacing, alignment, scale, contrast로 먼저 해결한다.
- 제품 UI에서는 마케팅 카피보다 유틸리티 카피를 우선한다.

## Current Fixed Intent

- UI/UX 단계는 완료된 것으로 간주한다.
- 현재 단계의 우선순위는 발표 대비 검증, DB 연동 확인, 로딩 최적화, 기능 안정화다.
- 메인페이지와 상세페이지를 포함한 확정 UI는 특별한 깨짐이 없는 한 유지한다.
- 한 페이지를 동시에 여러 방향으로 흔들지 않는다.

## Working Method

- 구현 전 현재 상태와 영향 범위를 먼저 읽는다.
- 발표 전에는 큰 리디자인보다 실제 동작, API 연결, 상태 흐름, 에러 처리부터 본다.
- DB 연동이 걸린 화면은 성공/실패/빈 상태까지 확인한다.
- 로딩이 긴 구간은 불필요한 중복 호출, N+1, 과한 렌더링부터 점검한다.
- 사용자가 직접 짚은 문제와 실제 시연 경로를 최우선으로 수정한다.

## Verification

- 성공이라고 말하기 전에 빌드나 실제 증거를 먼저 확인한다.
- 프론트 변경 후 기본 검증은 `npm run build`다.
- 발표 핵심 경로는 로그인, 리스트, 상세, 예약, 마이페이지, 판매자, 관리자 순으로 실제 흐름을 점검한다.
- 백엔드/DB 연동이 필요한 변경은 가능하면 실제 응답 기준으로 확인한다.
- 검증하지 못한 내용은 `UNVERIFIED`로 둔다.

## Shell Rule

- `rtk` 설치 경로는 `/opt/homebrew/bin/rtk`다.
- 가능한 쉘 명령은 `rtk` 우선 사용한다.
- 예: `rtk git status`, `rtk grep "pattern" src`, `rtk read file`, `rtk test npm run build`
- `rtk` 가 지원하지 않거나 실패하면 일반 명령으로 fallback 한다.
